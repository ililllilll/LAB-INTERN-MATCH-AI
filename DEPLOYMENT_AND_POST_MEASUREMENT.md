# 수정본 배포 및 성능 재검증 절차

## 1. 배포

이 폴더의 파일 전체를 기존 GitHub Pages 저장소 루트에 덮어씁니다. 기존 파일을 일부 남겨 두기보다 저장소의 현재 서비스 파일을 교체하는 방식이 안전합니다.

배포가 실제 반영된 한국시간을 반드시 기록합니다.

```text
수정본 실제 배포 시각(KST): YYYY-MM-DD HH:MM
```

이 시각은 이후 `--split-kst` 값으로 사용합니다.

## 2. 배포 직후 확인

1. 첫 화면에서 인턴, 분야별 연구실, 학교별 연구실로 이동
2. 각 학교에서 기존에 결과가 잘 나오던 검색 1개 확인
3. DGIST에서 `기계` 검색 후 결과 표시 확인
4. 알 수 없는 표현으로 0개가 발생할 때 분야 선택형 복구 패널 확인
5. 통계 동의와 거부 각각에서 검색 기능이 동일하게 작동하는지 확인
6. 공식 홈페이지 및 프로필 링크 확인

## 3. 추가 수집 기간

최소 7일, 가능하면 14일 동안 추천 알고리즘과 UI를 추가 변경하지 않습니다. 홍보를 추가했다면 날짜와 시간을 기록합니다.

우선 판단 조건:

- 수정 후 직접 입력 검색 100회 이상
- 수정 후 10자 이하 직접 입력 검색 30회 이상
- 자동 보강 적용 20회 이상이면 보강 성공률도 참고 가능

표본이 부족하면 기간을 연장합니다.

## 4. 수정 전 기준값

엄격한 공개 후 구간은 2026년 7월 10일 20:00부터 7월 12일 01:00 미만입니다.

- 직접 입력 결과 0개 비율: 38/197 = 19.29%
- 10자 이하 직접 입력 결과 0개 비율: 26/86 = 30.23%
- 대학원 전체 결과 0개 비율: 38/356 = 10.67%
- 대학원 공식 링크 클릭: 검색 100회당 16.85회

## 5. GoatCounter export 분석

수정 후 export ZIP을 내려받아 다음처럼 실행합니다.

```bash
python analysis/goatcounter_analysis.py goatcounter-export.zip \
  --output analysis_after \
  --timezone Asia/Seoul \
  --start-kst 2026-07-10T20:00 \
  --end-kst YYYY-MM-DDTHH:00 \
  --split-kst YYYY-MM-DDTHH:MM
```

`split-kst`는 실제 수정본 배포 시각입니다. GoatCounter가 시간 단위 집계이므로 전후를 엄격히 나누려면 배포 시각 다음 정각부터 수정 후 구간으로 잡는 것이 가장 안전합니다.

## 6. 최종 성과 판단

한 지표만으로 결론 내리지 않습니다. 다음 중 최소 두 항목이 같은 방향으로 개선되는지 봅니다.

1. 10자 이하 직접 입력 결과 0개 비율 감소
2. 전체 직접 입력 결과 0개 비율 감소
3. 자동 보강 성공률이 충분히 높음
4. 분야 선택 복구 후 결과 1개 이상 비율이 높음
5. 대학원 검색 100회당 공식 링크 클릭 증가
6. DGIST와 KAIST의 학교별 지표 개선

수정 전후 데이터는 무작위 A/B 실험이 아니므로 “유일한 원인임을 증명했다”가 아니라 “수정 이후 동일 지표가 개선되는 경향을 확인했다”고 표현합니다.
