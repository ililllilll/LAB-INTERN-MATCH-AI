# LAB MATCH AI 초기 이용 데이터 기반 검색 성능 개선 보고서

## 1. 분석 범위

- 에브리타임 게시: **2026년 7월 10일 19:36(KST)**
- GoatCounter export: **2026년 7월 12일 00:53(KST)**
- 엄격한 분석 구간: **2026년 7월 10일 20:00부터 7월 12일 01:00 미만**
- GoatCounter가 시간 단위로 집계하므로 19:36~19:59의 24분은 공개 전·후 행동을 정확히 분리할 수 없어 제외했습니다.
- 조회, 검색, 클릭은 사람 수가 아니라 익명 집계 이벤트 횟수입니다.

## 2. 관찰된 핵심 문제

### 문제 A. 대표 분야 버튼보다 직접 입력 검색의 실패가 컸음

| 검색 방식 | 검색 | 결과 0개 | 결과 0개 비율 |
|---|---:|---:|---:|
| 대표 분야 버튼 | 190회 | 0회 | 0.00% |
| 직접 입력 전체 | 197회 | 38회 | 19.29% |
| 직접 입력 10자 이하 | 86회 | 26회 | 30.23% |
| 직접 입력 31자 이상 | 59회 | 2회 | 3.39% |

대표 분야 버튼은 사전에 검증된 구체적 표현을 사용해 190회 모두 결과가 표시됐습니다. 반면 학생이 직접 짧게 입력한 검색은 약 3회 중 1회가 0개였습니다. 긴 문장 검색의 실패율은 3.39%였으므로, 데이터베이스 전체 부족보다는 **짧은 표현을 내부 연구 키워드로 연결하는 단계가 엄격한 것**이 더 유력합니다.

### 문제 B. 분류되지 않은 표현에서 실패가 집중됨

- 전체 결과 0개: **38/387회 (9.82%)**
- 익명 분야 `other`: **30/89회 (33.71%)**
- 대학원 검색의 `other`: **30/62회 (48.39%)**
- 전체 0개 검색 38회 중 `other`가 30회로 **78.9%**를 차지했습니다.

검색어 원문은 수집하지 않으므로 어떤 단어였는지는 알 수 없습니다. 따라서 원문 수집을 추가하는 대신 결과 0개가 발생했을 때 넓은 익명 분야를 선택하도록 하는 복구 기능이 필요했습니다.

### 학교별 참고 지표

| 학교/서비스 | 검색 | 결과 0개 비율 | 공식 링크 클릭/검색 100회 |
|---|---:|---:|---:|
| intern | 31회 | 0.00% | 22.58회 |
| field | 0회 | 0.00% | 0.00회 |
| dgist | 100회 | 22.00% | 18.00회 |
| snu | 99회 | 2.02% | 23.23회 |
| kaist | 110회 | 10.00% | 5.45회 |
| postech | 47회 | 6.38% | 27.66회 |


KAIST 공식 링크 클릭이 낮고 DGIST 결과 0개 비율이 높았지만, 학교 단위 지표만 수정하면 원인을 과도하게 단정할 수 있습니다. 반면 직접 입력·짧은 입력·`other`라는 세 지표가 같은 방향을 보여 **검색 실패 복구**를 1순위 개선 대상으로 선택했습니다.

## 3. 적용한 알고리즘 개선

### 3.1 원래 검색 우선, 실패 검색만 보강

1. 사용자가 입력한 원문으로 기존 추천 알고리즘을 먼저 실행합니다.
2. 결과가 1개 이상이면 기존 점수와 순서를 그대로 유지합니다.
3. 결과가 0개이고, 10자 이하이며, 하나의 고신뢰 분야로만 해석될 때 브라우저 안에서 관련 동의어를 추가해 한 번 재검색합니다.
4. 자동 보강 후에도 0개이면 넓은 분야 선택형 복구 패널을 표시합니다.

이 구조는 성공하던 검색까지 넓혀 오탐을 만드는 것을 방지합니다.

### 3.2 개인정보를 수집하지 않는 결과 0개 복구 UI

0개 결과 화면에서 반도체·전자, AI·데이터, 바이오·의료, 로봇·제어, 소재·화학, 에너지·환경 등의 분야를 선택할 수 있습니다. 선택한 분야는 브라우저 안에서 표준 연구 키워드로 바뀌며 검색어 원문은 GoatCounter에 전송되지 않습니다. 통계 수집을 거부해도 복구 기능은 동일하게 사용할 수 있습니다.

## 4. 추가한 정량 검증 이벤트

| 이벤트 | 측정 목적 |
|---|---|
| `lm-query-assist-applied-*` | 원래 검색 0개 후 자동 보강이 실제 적용된 횟수 |
| `lm-query-assist-outcome-*` | 자동 보강 후 표시 결과 수 구간 |
| `lm-zero-recovery-shown-*` | 자동 보강 이후에도 0개여서 복구 패널이 표시된 횟수 |
| `lm-zero-recovery-choice-*` | 사용자가 넓은 분야를 선택한 횟수 |
| `lm-zero-recovery-outcome-*` | 분야 선택 후 결과가 회복됐는지 |
| `lm-zero-recovery-external-*` | 회복된 결과에서 공식 홈페이지·프로필을 클릭한 횟수 |

이벤트에는 페이지, 넓은 분야, 입력 길이 구간, 결과 수 구간만 포함합니다. 검색어 원문, 교수명, 사용자 ID, 개인별 경로는 포함하지 않습니다.

## 5. 수정 전 기준값과 향후 성공 판정

| 1차 KPI | 수정 전 기준값 |
|---|---:|
| 직접 입력 전체 결과 0개 비율 | 19.29% (38/197) |
| 10자 이하 직접 입력 결과 0개 비율 | 30.23% (26/86) |
| 대학원 전체 결과 0개 비율 | 10.67% (38/356) |
| 대학원 공식 링크 클릭/검색 100회 | 16.85회 (60/356) |

수정 후에는 같은 요일 구성을 포함해 최소 7일, 가능하면 14일을 수집합니다. 10자 이하 직접 입력 검색이 최소 30회 이상 쌓인 뒤 다음을 비교합니다.

1. 10자 이하 직접 입력 결과 0개 비율의 %p 변화
2. 자동 보강 성공률
3. 결과 0개 복구 선택률과 복구 성공률
4. 복구 성공 100회당 공식 링크 클릭
5. 학교별 결과 0개 비율과 공식 링크 클릭의 동반 변화

**아직 수정 후 실제 이용 데이터가 없으므로 개선 효과가 달성됐다고 쓰면 안 됩니다.** 현재 확인된 성과는 데이터를 세분화해 실패 조건을 찾고, 해당 조건만 보정하며, 효과를 다시 측정할 체계를 구현한 것입니다.

## 6. 기능 검증

- JavaScript 문법 검사: 수정된 `analytics.js`, `query-assist.js`, 다섯 연구실 추천 `app.js` 통과
- 회귀 확인: 서울대 `운영체제` 검색은 수정 전후 모두 5개 카드, 1위 엄현상 교수님으로 유지
- 실패 복구 확인: DGIST `기계` 검색은 수정 전 0개에서 수정본에서는 기계·로봇·제어 관련 결과가 표시됨
- 통계 거부 시에도 검색 복구 UI가 작동하도록 기능과 수집 동의를 분리
- 분석기 보강: 한국시간 필터, `/index.html` 경로 정규화, `lm-reset-*`, 신규 복구 이벤트, 미파싱 이벤트 보고 추가

## 7. SK하이닉스 양산기술 연결

> 전체 검색 평균만 보지 않고 입력 방식과 길이, 학교, 익명 분야, 결과 수로 데이터를 분해했습니다. 대표 분야 검색은 실패가 없었지만 10자 이하 직접 입력은 결과 0개 비율이 30.23%임을 확인해, 데이터베이스 전체가 아니라 짧은 표현을 연구 키워드로 연결하는 조건을 이상 구간으로 특정했습니다. 정상 검색의 결과는 유지하고 실패한 조건에서만 동의어 보강과 분야 선택 복구를 적용했으며, 이후 보강 성공률과 공식 링크 이동을 같은 집계 지표로 재검증하도록 설계했습니다. 이는 양산 현장에서 전체 수율이 아니라 공정·장비·조건별 데이터를 분해해 이상 조건만 수정하고 전후 효과를 검증하는 방식과 연결됩니다.
